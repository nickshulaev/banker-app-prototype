# Задача: добавить онбординг-флоу в прототип Banker (изолированно, не сломав текущий апп)

## Цель
Добавить в прототип **пре-авторизационный флоу входа/онбординга** как **отдельную фазу приложения**,
не трогая логику уже авторизованного аппа. Ключевое требование — **нулевая регрессия**: при
выключенном онбординге прототип работает ровно как сейчас.

## ⚠️ Механика (главное — единый вход, ветки = исходы гейтвеев)
Вход **всегда один — ввод номера телефона**. Ветки это НЕ отдельные точки входа, а **терминалы
роутера**, в которые ведут решения-гейтвеи. По выверенной BPMN (lane «Freedom KZ»):

```
Старт → [phone_entry] Ввод номера телефона → Продолжить
        └─ gateway «Номер в БД?»
            ├─ ДА → [otp_login] OTP
            │        └─ gateway «Демо-юзер?»
            │            ├─ нет → [authorize] Авторизация ──▶ ВХОД В АПП (onComplete)
            │            └─ да  → подпроцесс A: Онбординг (начало/продолжение)
            └─ нет → [iin_stage] Этап ввода ИИН
                     ├─ «Ввести ИИН» → [iin_entry] Ввод ИИН
                     │        └─ gateway «Юзер с таким ИИН есть?»
                     │            ├─ ДА  → [otp_changenum] OTP ──▶ подпроцесс B: Смена номера
                     │            └─ нет → подпроцесс C: eResidency
                     └─ «У меня нет ИИН» → подпроцесс C: eResidency (редирект)
```

Терминалы (handoff'ы):
- **Авторизация** — номер в БД, не демо → вход в приложение.
- **Подпроцесс A (онбординг-профиль)** — номер в БД, демо-юзер → выпуск карты + AML-анкета + адрес + Sumsub.
- **Подпроцесс B (смена номера)** — номера нет в БД, но ИИН существует → форма смены + Sumsub+Q19.
- **Подпроцесс C (eResidency)** — ИИН нет / не найден → редирект на eResidency, возврат ко входу.

«Восстановление доступа» отдельной веткой НЕ делаем — выбран **Вариант 2** (единый вход по телефону);
этот сценарий = путь «номер в БД → OTP → авторизация».

## Архитектура текущего прототипа (почему интегрируем именно так)
- `freedom-v6.jsx` — один файл (~6800 строк). Корень `FreedomV6` держит `theme`, `locked`,
  `activeTab`, `navStack` (мини-роутер аппа), `sheet`.
- Приложение стартует с `locked=true` → `LockScreen` накрывает оверлеем; разлок → апп. Паттерн
  «полноэкранный поток поверх аппа, гейт по флагу» уже есть.
- `navStack` — роутер **уже авторизованного** аппа. Онбординг идёт ДО аппа → на `navStack` его НЕ вешать.

## Принципы изоляции (НЕ нарушать)
1. **Весь онбординг — в новом файле `onboarding.jsx`.** В `freedom-v6.jsx` диф минимальный
   (импорт + 1 стейт + 1 mount + 1 секция дебага). Контент шагов в `freedom-v6.jsx` НЕ добавлять.
2. Онбординг получает только `C` (палитра темы), `initial` (для дебага) и `onComplete`. Никакого
   доступа к `navStack`, продуктам, `FEATURE_FLAGS`, `blockVis`.
3. **Не переименовывать/не рефакторить** существующие компоненты. Визуальные примитивы
   переиспользовать копированием стиля: OTP-клавиатура — как `LockScreen`, поля — как
   `FormField`/`ScreenShell`, статус-бар — `StatusBar`, нажатия — `data-press`.
4. Без новых зависимостей (никакого react-router). Свой мини-роутер внутри `onboarding.jsx`.

## Файл 1 (новый): `onboarding.jsx`

### 1.1 Модель: единый роутер + гейты
Стартовый шаг ВСЕГДА `phone_entry`. Решения берутся из **мок-флагов гейтов** (их выставляет дебаг,
чтобы из реального старта дойти до любого терминала):

```js
// мок-ответы гейтвеев (выставляются из дебаг-меню)
const defaultGates = { numberInDb: false, isDemoUser: false, iinExists: false };

// шаги-роутера + переходы (next может быть функцией от gates/formData)
export const ONB_STEPS = [
  { key: "phone_entry", title: "Ввод номера телефона",
    next: (g) => g.numberInDb ? "otp_login" : "iin_stage" },
  { key: "otp_login", title: "Код-пароль (OTP)",
    next: (g) => g.isDemoUser ? "A_card_select" : "authorize" },
  { key: "authorize", title: "Авторизация", terminal: "app" }, // → onComplete()
  { key: "iin_stage", title: "Этап ввода ИИН" }, // выбор: «Ввести ИИН» | «У меня нет ИИН»
  { key: "iin_entry", title: "Ввод ИИН",
    next: (g) => g.iinExists ? "otp_changenum" : "C_eres_intro" },
  { key: "otp_changenum", title: "Код-пароль (OTP)", next: () => "B_chg_form" },
  // подпроцессы A/B/C — см. 1.3
];
```

Локальный стейт `OnboardingFlow`: `stepKey`, `gates`, `formData`, `verifyState`
(`idle|checking|success|rejected|pending` для Sumsub). Навигация: `goNext()` (читает `gates`),
`goBack()`, `jumpTo(stepKey)`, `setGate(name, val)`.

### 1.2 Общие под-компоненты
- **`PhoneEntry`** — ввод номера телефона + «Продолжить». Дисклеймеры из дизайна: «застрахованы до
  ₸20 млн (~$45 000)», «Продолжая… не являетесь ПДЛ… FATCA».
- **`OtpKeypad`** — «Код-пароль», 6 цифр, цифровая клавиатура, таймер «Доступно через…», «Повторить».
  Стиль — как `LockScreen`. Используется и для `otp_login`, и для `otp_changenum`.
- **`IinStage` / `IinEntry`** — экран развилки («Ввести ИИН» / «У меня нет ИИН») и экран ввода ИИН
  (12 цифр, подсказка «Где найти ИИН»: РК — удостоверение/eGov, иностранцы — через eResidency).
- **`SumsubFlow`** (под-машина, общая для A и B): `intro` («Подтверждение личности, ~2 мин»: личный
  документ / селфи / адрес) → `selfie` («Расположите лицо внутри рамки») → `checking` («Получили
  видео, начали проверку», «Обновить»/«Выйти») → терминалы `success` / `rejected` («Проверка
  отклонена» → «Обратиться в поддержку»/«Начать заново») / `pending` («Сообщим, когда подтвердим»).
  Подпись «Основано на технологии Sumsub».
- **`SuccessScreen`** — «Поздравляем!» + CTA «Перейти в приложение» → `onComplete`.

### 1.3 Подпроцессы (downstream-терминалы)

**A. Онбординг-профиль** (демо-юзер; «Начало/продолжение онбординга»):
`A_card_select` (Выпустить/активировать карту: Invest / Deposit) →
`A_aml` (**AML-анкета целиком, все 22 вопроса** — см. [`docs/aml-questionnaire-spec.md`](aml-questionnaire-spec.md);
перенести `AML_QUESTIONS` в `onboarding.jsx`, рендер обобщённый по `type`, условные поля Q02/Q08/Q16/Q01/Q13) →
`A_address` (экран «Адрес»: страна рождения/регистрации/проживания, тумблер «совпадает с регистрацией») →
`A_sumsub` (`SumsubFlow`) → `A_success` → `onComplete`.

**B. Смена номера** (ИИН существует; «Запуск процесса смены номера»):
`B_chg_form` (старый номер, ИИН, новый номер; дисклеймер согласия) →
`B_statement` («Заявление на изменение анкетных данных», превью, «Отправить»/«Закрыть») →
`B_sumsub` (`SumsubFlow` — Sumsub + Q19 на подтверждение номера) →
`B_processing` («Смена номера телефона — В процессе», может висеть до 1 суток, пускает в приложение;
CTA «Перейти в приложение» → `onComplete`) →
(опц.) `B_success_push` (макет пуша «Вы успешно сменили номер на …»).

**C. eResidency** (ИИН нет / не найден; «Редирект на eResidency»):
`C_eres_intro` («Получите eResidency Казахстана»: 1) оформите онлайн — ID-карта, ИИН, eSIM;
2) войдите по номеру eSIM. CTA «Получить eResidency») → мок-редирект (внешняя заглушка) →
возврат на `phone_entry` (после получения eSIM пользователь входит как обычно).

### 1.4 Дебаг-навигация (вместо «прыжка на ветку»)
Старт всегда с `phone_entry`. Дебаг управляет **гейтами** (чтобы дойти до любого терминала из
реального входа) + даёт быстрые пресеты-персоны:
- Тумблеры гейтов: «Номер в БД», «Демо-юзер», «ИИН существует».
- Пресеты-персоны (выставляют гейты разом): «Существующий клиент (вход)» (numberInDb=1, demo=0) ·
  «Демо/онбординг» (numberInDb=1, demo=1) · «Смена номера» (numberInDb=0, iinExists=1) ·
  «eResidency» (numberInDb=0, iinExists=0).
- (Опц.) джампер `jumpTo` по `ONB_STEPS` — для осмотра конкретного экрана.
Экспортировать `ONB_STEPS` и список пресетов для рендера в `BottomSheet`.

### 1.5 Сигнатура
```jsx
export default function OnboardingFlow({ C, initial, onComplete }) { ... }
// initial = { gates?, stepKey? }; по умолчанию stepKey="phone_entry"
// onComplete() = вход/онбординг завершён → апп показывает главный экран
```

## Файл 2: `freedom-v6.jsx` (минимальные правки)
1. `import OnboardingFlow, { ONB_STEPS } from "./onboarding.jsx";`
2. Стейт фазы (дефолт выключен — нулевая регрессия):
   ```js
   const [onb, setOnb] = useState(null); // null = выкл; иначе { gates, stepKey }
   ```
3. Mount рядом с `LockScreen` (конец JSX):
   ```jsx
   {onb && (
     <OnboardingFlow C={C} initial={onb}
       onComplete={() => { setOnb(null); setLocked(false); setActiveTab("products"); }} />
   )}
   ```
4. В `BottomSheet` — секция **«Онбординг»**: тумблер вкл/выкл (старт с `phone_entry`), тумблеры гейтов
   (Номер в БД / Демо / ИИН существует) и пресеты-персоны. Прокинуть `onb`, `setOnb` из корня.

## Не делать
- Не делать отдельные точки входа на ветки — **вход один: `phone_entry`**, ветки = исходы гейтов.
- Не вешать онбординг на `navStack`; не менять логику `LockScreen`/`MainScreen` (только mount рядом).
- Не подключать react-router; не добавлять контент шагов в `freedom-v6.jsx`.
- Не называть антифрод/комплаенс в клиентских текстах (AML — сбор данных, формулировки нейтральные).

## Проверка (acceptance)
1. `npm run build` — без ошибок.
2. **Регрессия:** при `onb === null` прототип идентичен прежнему.
3. Онбординг всегда **стартует с `phone_entry`** (ввод номера). Дебаг-гейты/пресеты задают маршрут.
4. Маршруты-роутера доходят до правильных терминалов:
   - numberInDb=1, demo=0 → OTP → **Авторизация** → вход в апп.
   - numberInDb=1, demo=1 → OTP → **подпроцесс A** (карта → AML 22 вопроса → адрес → Sumsub → success → апп).
   - numberInDb=0, «Ввести ИИН», iinExists=1 → OTP → **подпроцесс B** (форма → заявление → Sumsub → «В процессе» → апп).
   - numberInDb=0, iinExists=0 (или «У меня нет ИИН») → **подпроцесс C** (eResidency → редирект → возврат на `phone_entry`).
5. AML-анкета — **все 22 вопроса**, 6 секций, условные поля работают (Q02→файл, Q08→банки,
   Q16→стоимость недвижимости, Q01/Q13 «Другое»→текст, Q13 «Показать ещё»).
6. `SumsubFlow` показывает все состояния: intro → selfie → checking → success/rejected/pending.
7. Тема (`stripe`/`dark`) применяется через `C`.
8. `freedom-v6.jsx` изменён минимально; вся начинка — в `onboarding.jsx`.
